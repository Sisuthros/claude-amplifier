## Problem statement

The user research (`docs/research/claude-user-pains-2026-05-21.md`, ~2100 words) ranked Claude users' top 10 problems. Two clusters dominate impact and frequency:

1. **Memory & Continuity** — Claude forgets prior sessions; CLAUDE.md instructions get ignored; auto-compact silently destroys work in progress.
2. **Reliability & Compliance** — hallucinated facts get persisted into memory and then re-asserted as truth (Issue #27430, the "confabulation feedback loop"); subagents explode token budgets; MCP tools collide.

Existing tools (`claude-mem`, `MemCP`, Obsidian Memory MCP, OpenMemory, mem0) all address Cluster 1 reasonably well. **None** address Cluster 2 — verification, evidence, and pattern-based prevention. This is the market gap.

Amplifier 1.3.0 already provides persistent memory, decision tracking, and `pattern_key` aggregation. 1.4.0 turns that foundation into a **verification + prevention engine** — the only Claude memory MCP that refuses to spread hallucinations and warns before known failure patterns repeat.

## Headline value proposition

> **The only Claude memory MCP that prevents repeat failures and refuses to persist hallucinations.**

Two pillars:

1. **Pattern Oracle (`amplify_preflight`)** — given a prompt and project context, returns a risk assessment based on stored lessons, decisions, and patterns. Warns the user (or Claude) before known failure modes recur. A data-driven, personal version of Karpathy's 220k-star "preflight checklist" CLAUDE.md trick.
2. **Verification-Gated Memory** — every lesson and decision has a `verification_status` (`claim` | `evidence` | `confirmed`). Claims need evidence (git commit, test run, user confirmation) before they are treated as facts. Pattern Oracle weighs `confirmed` facts ten times more than unverified claims.

These two features reinforce each other. The Oracle is only useful if its data is trustworthy, and the verification gate is only useful if its data gets consulted at decision time. They ship together.

## Non-goals (YAGNI)

- **No automatic git-history scanning** for evidence — verification is explicit, not inferred.
- **No vector-embedding semantic search** — sticks with SQLite + simple keyword + pattern_key matching. (Semantic search is a 1.5.0 candidate, not 1.4.0.)
- **No cloud sync, no telemetry, no team mode** — local-first remains the core property.
- **No "Claude-only" coupling** — the MCP surface stays portable. Cursor, Cline, Continue, RooCode, Aider can all call the same tools.
- **No migration script** — 1.3.x data reads as if every existing lesson is `confirmed`. No data loss, no manual intervention.

## MCP tool surface (5 new tools)

### `amplify_preflight`

```jsonc
{
  "name": "amplify_preflight",
  "description": "Before starting a task, check for matching failure patterns and past lessons. Returns risk assessment and suggested approach.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "project":  { "type": "string", "description": "Project name" },
      "prompt":   { "type": "string", "description": "The prompt or task description the user/agent is about to execute" },
      "context":  { "type": "string", "description": "Optional: surrounding context (file names, recent commits, etc.)" }
    },
    "required": ["project", "prompt"]
  }
}
```

**Response shape:**

```ts
{
  risk_level: "low" | "medium" | "high" | "critical",
  matched_patterns: Array<{
    pattern_key: string,
    title: string,
    frequency: number,
    last_seen: string,         // ISO date
    verification_status: "claim" | "evidence" | "confirmed",
    confidence: number          // 0..1
  }>,
  matched_lessons: Array<{ id: number, title: string, type: string, severity: string }>,
  matched_decisions: Array<{ id: number, title: string, category: string }>,
  suggested_approach: string,   // human-readable advice
  evidence_quality: "strong" | "weak" | "anecdotal"
}
```

**Risk calculation:**

```
score = Σ (pattern.frequency * pattern.confidence * weight(verification_status))
        for each matched pattern

weight(verification_status):
  confirmed → 1.0
  evidence  → 0.6
  claim     → 0.2

risk_level:
  score < 1.0           → low
  1.0 ≤ score < 3.0     → medium
  3.0 ≤ score < 6.0     → high
  score ≥ 6.0           → critical
```

### `amplify_record_claim`

Replaces 1.3.0's bare `amplify_learn`. All new lessons default to `claim`. Aliased so 1.3.x callers still work; `amplify_learn` becomes a thin wrapper that calls `amplify_record_claim`.

```jsonc
{
  "name": "amplify_record_claim",
  "description": "Record a lesson as a claim (not yet verified). Requires evidence before it counts as fact. Use this for any 'I just learned X' moment.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "project":    { "type": "string" },
      "type":       { "enum": ["mistake", "success", "insight", "warning"] },
      "title":      { "type": "string" },
      "description":{ "type": "string" },
      "pattern_key":{ "type": "string" },
      "initial_confidence": { "type": "number", "minimum": 0, "maximum": 1, "default": 0.5 }
    },
    "required": ["project", "title", "description"]
  }
}
```

### `amplify_verify_claim`

```jsonc
{
  "name": "amplify_verify_claim",
  "description": "Promote a claim to evidence or confirmed_fact by linking concrete evidence.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "id":              { "type": "number" },
      "evidence_type":   { "enum": ["git_commit", "test_run", "user_confirmation", "external_doc", "manual_review"] },
      "evidence_link":   { "type": "string", "description": "Git SHA, test ID, URL, or note" },
      "promote_to":      { "enum": ["evidence", "confirmed"], "default": "evidence" }
    },
    "required": ["id", "evidence_type", "evidence_link"]
  }
}
```

Promotion rules:
- `claim` + 1 evidence → `evidence` (confidence ↑ to 0.7)
- `evidence` + 1 user_confirmation OR 2 distinct evidence types → `confirmed` (confidence = 1.0)
- A lesson can be demoted back to `claim` if all its evidence_links are removed.

### `amplify_promote_pattern`

```jsonc
{
  "name": "amplify_promote_pattern",
  "description": "Promote a pattern_key from per-project to global when it has fired in 2+ projects.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "pattern_key": { "type": "string" },
      "scope":       { "enum": ["global", "project"], "default": "global" }
    },
    "required": ["pattern_key"]
  }
}
```

Auto-trigger: when a `pattern_key` appears in lessons from 2 or more projects AND has at least one `confirmed` lesson, the next call to `amplify_record_claim` or `amplify_preflight` proposes promotion. The user (or Claude) calls this tool to accept.

### `amplify_evidence_chain`

```jsonc
{
  "name": "amplify_evidence_chain",
  "description": "Return the full evidence chain that supports a stored fact. Useful for auditing why Amplifier 'knows' something.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "id":   { "type": "number" },
      "kind": { "enum": ["lesson", "decision"], "default": "lesson" }
    },
    "required": ["id"]
  }
}
```

## Data model changes

```sql
-- lessons table (additive)
ALTER TABLE lessons ADD COLUMN verification_status TEXT DEFAULT 'confirmed';
  -- existing rows = confirmed (backwards compatible)
  -- new rows from amplify_record_claim = claim
ALTER TABLE lessons ADD COLUMN evidence_links TEXT DEFAULT '[]';
  -- JSON array of {evidence_type, evidence_link, recorded_at}
ALTER TABLE lessons ADD COLUMN confidence REAL DEFAULT 1.0;

-- same triple on decisions
ALTER TABLE decisions ADD COLUMN verification_status TEXT DEFAULT 'confirmed';
ALTER TABLE decisions ADD COLUMN evidence_links TEXT DEFAULT '[]';
ALTER TABLE decisions ADD COLUMN confidence REAL DEFAULT 1.0;

-- new table: pattern promotions (audit log)
CREATE TABLE IF NOT EXISTS pattern_promotions (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  pattern_key  TEXT NOT NULL,
  promoted_at  TEXT NOT NULL,
  promoted_from_projects TEXT NOT NULL,  -- JSON array
  total_frequency INTEGER NOT NULL
);
```

Migration runs once on first 1.4.0 launch, idempotent via `PRAGMA user_version`. No downgrade path needed because 1.3.x ignores the new columns.

## Error handling

| Failure mode | Behaviour |
|--------------|-----------|
| Preflight called on empty database | Return `risk_level: "low"`, empty matches, advice "no history yet — proceed with normal caution" |
| Verify claim with invalid evidence type | Return error string; do not change DB state |
| Pattern promotion attempted but pattern only in 1 project | Return error explaining the rule; do not promote |
| Migration fails partway | Roll back the failed `ALTER`; preserve previous schema; log to stderr; continue with reduced functionality (1.3.0-compatible mode) |
| SQLite locked (WAL contention) | Retry up to 3 times with 50ms backoff before erroring |

## Risks and mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Pattern Oracle has too many false positives, becomes "the boy who cried wolf" | Medium | Start with conservative thresholds (score ≥ 3.0 = high). Tunable via env var `AMPLIFIER_ORACLE_THRESHOLD`. |
| Users don't bother calling `amplify_verify_claim` → everything stuck as claim | Medium | Default user-confirmation evidence type can be triggered by a quick "yes that worked" message. Document the workflow in README. |
| Migration corrupts an existing 1.3.x DB | Low | All ALTERs are additive; user_version bumps only after success. WAL mode means the bad transaction rolls back cleanly. |
| Tool surface becomes too crowded (8 tools now in 1.4.0) | Medium | Group related tools in description. Plan for 1.5.0 consolidation if telemetry shows confusion. |
| Suomi/English language mix in pattern_keys breaks matching | Low | pattern_key is opaque ASCII (kebab-case), already enforced. Other fields are language-agnostic. |
