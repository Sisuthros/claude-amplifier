Persistent memory MCP for Claude with Pattern Oracle preflight + verification-gated lessons.

Stop re-explaining the same architectural decisions to Claude every session. Stop letting Claude's unverified guesses quietly poison future advice.

The problem

Claude forgets. Claude also remembers things it shouldn't.

  • “Get some sleep” — at 6 PM. The previous session had run 02:00–06:00. Claude assumed the conversation was continuous. No clock check, no session boundary, no idea you'd just come home from work.
  • MongoDB suggestions, four times this week. We switched to Postgres three months ago. The decision and its rationale lives in your head — nowhere Claude can read it.
  • The same bug, three different sessions. Three nearly-identical mistakes in code review. Claude has no memory that any of them happened.
  • A guess, now a “lesson.” Claude inferred a fix that wasn't actually correct, wrote it down, and now that guess feeds back into every future session as if it were verified. (issue #27430)

The solution

An MCP server with a local SQLite database, a Pattern Oracle that warns before Claude walks into a known landmine, and a verification chain that distinguishes claimed from evidenced from confirmed.

1. Pattern Oracle — preflight risk check

> amplify_preflight({ task: "Configure agent endpoint" })

⚠  HIGH RISK (score 4.20)
Evidence quality: STRONG

Matched patterns (3):
  • [confirmed] Avoid model names containing another provider's prefix
    seen 3x across 2 projects, severity: critical
  • [confirmed] Read provider /v1/models before configuring fallback chains
    seen 5x across 3 projects, severity: high
  • [evidence]  Heartbeat models need TPM ≥ 30k
    seen 2x, severity: high

Suggested approach: Read your runtime's model-routing docs before
choosing the model string. Verify against `GET /v1/models`.

2. Verification-gated memory — no more confabulation feedback loop

// Claude infers a fix it hasn't verified
> amplify_record_claim({ ... })            status: claim     conf: 0.5

// Build passes, attach evidence
> amplify_verify_claim({
    id: 17,
    evidence_type: "build_passed",
    evidence_link: "https://github.com/.../actions/runs/12345"
  })                                         status: evidence  conf: 0.7

// User confirms in the next session
> amplify_verify_claim({
    id: 17,
    evidence_type: "user_confirmation"
  })                                         status: confirmed conf: 1.0

The Oracle weights matches by status: a confirmed lesson at score 1.0 counts five times as much as a raw claim at 0.2. Unverified guesses cannot drown out hard-won truth.

vs. the official memory server

Feature server-memory claude-amplifier
Decisions with rationale + alternatives — ✓
Recurrence tracking (frequency ×N) — ✓
Pattern grouping (pattern_key) — ✓
Lifecycle: outcome check-ins / restore steps — ✓
Preflight risk warnings (Pattern Oracle) — ✓ v1.4.0
Verification-gated memory (claim / evidence / confirmed) — ✓ v1.4.0

The official server is great for “remember entities and how they relate.” Claude Amplifier is for “remember why we decided this, what keeps going wrong, and when I scheduled a follow-up.”

Quick start

npm install -g claude-amplifier@1.4.0
claude-amplifier init       # auto-detect Claude Desktop / Claude Code
claude-amplifier seed       # plant three starter lessons
# restart Claude. that's it.

Then add one line to your CLAUDE.md:

At the START of every session:
amplify_context_load({ project: "<your-project>", types: ["all"] })
